7 Ways to Detect People With Just Your MacBook

2026-03-28 · Series: WiFi Sensing

No ESP32, no Raspberry Pi, no extra hardware. Just your laptop and Python.

This is the companion piece to WiFi Can See You Move, which covers the full landscape including dedicated hardware. This article focuses exclusively on what you can do with a stock MacBook.

The situation

You want to detect whether someone is in a room, in your house, or near your computer. Maybe it's for home automation. Maybe it's a security project. Maybe you're just curious what your WiFi adapter can actually do.

The good news: your MacBook has a WiFi adapter, a Bluetooth radio, and a full network stack. That's enough for six out of seven detection methods. The bad news: the one method that actually works best (CSI extraction) is the one Apple's locked you out of. Here's everything else, with working code.

1. ARP / ping scanning

The simplest approach. If someone's phone is on your WiFi, they're home. Send ARP requests to every IP on your subnet and see who answers.

What you're actually doing

ARP (Address Resolution Protocol) maps IP addresses to MAC addresses on a local network. When you send an ARP request for 192.168.1.42, the device at that IP responds with its MAC address. By sweeping the whole subnet, you discover every active device.

This tells you "a device is connected" not "a person is present." Someone can leave their phone charging and go out. A smart TV is always connected. You need to know which MAC addresses belong to people's personal devices and filter accordingly.

Working code

#!/usr/bin/env python3
"""ARP presence scanner. Requires: pip3 install scapy"""
import time, json
from datetime import datetime
from scapy.all import ARP, Ether, srp

# Map MAC addresses to people (fill in your own)
KNOWN_DEVICES = {
    "aa:bb:cc:dd:ee:f1": "Alice iPhone",
    "aa:bb:cc:dd:ee:f2": "Bob Pixel",
    "aa:bb:cc:dd:ee:f3": "Alice MacBook",
}
SUBNET = "192.168.1.0/24"
INTERVAL = 30  # seconds between scans

def scan(subnet):
    """Return set of (ip, mac) tuples for responding devices."""
    ans, _ = srp(
        Ether(dst="ff:ff:ff:ff:ff:ff") / ARP(pdst=subnet),
        timeout=3, verbose=False
    )
    return {(r.psrc, r.hwsrc) for _, r in ans}

def main():
    print(f"Scanning {SUBNET} every {INTERVAL}s...")
    prev_people = set()
    while True:
        devices = scan(SUBNET)
        current_people = set()
        for ip, mac in devices:
            mac_lower = mac.lower()
            if mac_lower in KNOWN_DEVICES:
                current_people.add(KNOWN_DEVICES[mac_lower])

        arrived = current_people - prev_people
        departed = prev_people - current_people
        ts = datetime.now().strftime("%H:%M:%S")

        if arrived:
            print(f"[{ts}] ARRIVED: {', '.join(arrived)}")
        if departed:
            print(f"[{ts}] LEFT:    {', '.join(departed)}")
        if not arrived and not departed:
            who = ', '.join(current_people) if current_people else "nobody"
            print(f"[{ts}] Present: {who} ({len(devices)} devices total)")

        prev_people = current_people
        time.sleep(INTERVAL)

if __name__ == "__main__":
    main()

done

Limitations

  • Only detects devices actively connected to your WiFi
  • Phone in airplane mode = invisible
  • Guest's phone not on your WiFi = invisible
  • Can't distinguish "phone is here" from "person is here"
  • Some devices don't respond to ARP when in sleep mode
  • Scapy requires root on macOS: sudo python3 scanner.py

2. RSSI monitoring

Your MacBook constantly measures the signal strength of its WiFi connection. That number (RSSI, in dBm) changes when the physical environment between your laptop and router changes. When you walk between them, signal drops. When you leave, it recovers. If you track the fluctuation pattern, you can detect movement.

How sensitive is it?

In a typical home setup with laptop 4-5 meters from the router, a person walking through the line of sight causes a 3-8 dBm dip lasting 1-3 seconds. Someone sitting down in the same room causes a sustained 1-3 dBm offset. These numbers are small but statistically distinguishable from the normal 0.5-1 dBm noise floor if you accumulate enough samples.

The standard deviation of RSSI over a sliding window is the key metric. In an empty room, stdev stays below 1.0 dBm. When someone moves through, it spikes to 2.0-5.0+. A threshold of 1.5-2.0 catches most real movement while ignoring noise.

Working code

#!/usr/bin/env python3
"""RSSI motion detector for macOS.
Uses the airport utility to read signal strength.
No additional packages needed."""
import subprocess, time, statistics, sys
from collections import deque

AIRPORT = ("/System/Library/PrivateFrameworks/Apple80211.framework"
           "/Versions/Current/Resources/airport")
WINDOW = 20        # samples in sliding window
POLL_HZ = 2        # readings per second
MOTION_THRESH = 2.0 # stdev threshold for motion
IDLE_THRESH = 1.2   # stdev threshold to return to idle

def get_rssi():
    """Read current RSSI from airport utility."""
    try:
        r = subprocess.run([AIRPORT, "-I"],
                           capture_output=True, text=True, timeout=5)
        for line in r.stdout.splitlines():
            if "agrCtlRSSI" in line:
                return int(line.split(":")[1].strip())
    except Exception:
        pass
    return None

def get_ssid():
    """Read current SSID."""
    try:
        r = subprocess.run([AIRPORT, "-I"],
                           capture_output=True, text=True, timeout=5)
        for line in r.stdout.splitlines():
            stripped = line.strip()
            if stripped.startswith("SSID:"):
                return stripped.split(":", 1)[1].strip()
    except Exception:
        pass
    return "unknown"

def main():
    ssid = get_ssid()
    print(f"Monitoring RSSI on '{ssid}' at {POLL_HZ} Hz")
    print(f"Window: {WINDOW} samples | Motion threshold: {MOTION_THRESH} dBm stdev")
    print("-" * 60)

    buf = deque(maxlen=WINDOW)
    state = "IDLE"
    motion_count = 0
    idle_count = 0

    while True:
        rssi = get_rssi()
        if rssi is None:
            print("  [no signal]")
            time.sleep(1.0 / POLL_HZ)
            continue

        buf.append(rssi)

        if len(buf) >= 8:
            mean = statistics.mean(buf)
            sd = statistics.stdev(buf)
            rng = max(buf) - min(buf)

            # State machine with hysteresis
            if state == "IDLE" and sd > MOTION_THRESH:
                motion_count += 1
                if motion_count >= 3:  # debounce: 3 consecutive readings
                    state = "MOTION"
                    motion_count = 0
            elif state == "MOTION" and sd < IDLE_THRESH:
                idle_count += 1
                if idle_count >= 5:  # slower to go idle (avoid flicker)
                    state = "IDLE"
                    idle_count = 0
            else:
                if state == "IDLE":
                    motion_count = 0
                else:
                    idle_count = 0

            bar = "#" * int(sd * 5)
            print(f"  RSSI {rssi:4d} dBm | mean {mean:6.1f} | "
                  f"sd {sd:5.2f} | range {rng:2d} | "
                  f"{state:6s} {bar}")
        else:
            print(f"  RSSI {rssi:4d} dBm | buffering ({len(buf)}/{WINDOW})...")

        time.sleep(1.0 / POLL_HZ)

if __name__ == "__main__":
    main()

Tuning tips

  • Place your MacBook so the router is on the other side of the room you're monitoring. Person walks between them = maximum signal disruption.
  • 2.4 GHz is more sensitive to body movement than 5 GHz (longer wavelength, more diffraction around obstacles). If your router supports both, force your Mac to the 2.4 GHz network for this.
  • The threshold needs calibration per environment. Run for 5 minutes in an empty room, note the max stdev, set your threshold 50% above that.
  • Microwave ovens operating on 2.4 GHz will cause massive RSSI spikes. Account for this or switch to 5 GHz.

Limitations

  • Only detects movement near the line between your laptop and router
  • Cannot detect stationary presence (someone sitting still)
  • High false positive rate compared to CSI-based methods
  • Environmental noise (HVAC, appliances, other WiFi traffic) creates baseline drift
  • Single RSSI number vs. 64 subcarriers in CSI = massively less information

3. Probe request sniffing

WiFi devices periodically broadcast probe requests: "Hey, is MyHomeNetwork nearby?" Your MacBook's WiFi adapter can listen for these if you put it in monitor mode. Each probe contains the device's MAC address and sometimes the network name it's looking for.

The macOS caveats

This is where things get complicated on Mac. Modern macOS severely limits monitor mode. The built-in airport utility can capture in monitor mode, but:

  • It disconnects you from WiFi while monitoring (you can't be connected AND in monitor mode simultaneously)
  • Apple's newer chips (M1/M2/M3/M4) have inconsistent monitor mode support
  • You need to run as root
  • Capture quality varies by macOS version
# Enable monitor mode (disconnects WiFi)
sudo airport en0 sniff 6   # channel 6

# This creates a pcap file in /tmp/
# Parse it after capture:
tcpdump -r /tmp/airportSniffXXXXXX.cap \
    -e 'type mgt subtype probe-req' \
    -l 2>/dev/null | \
    grep -oE '(SA|DA):[0-9a-f:]+' | sort -u

The bigger problem: MAC randomization. Since iOS 14 and Android 10, devices randomize their MAC address in probe requests. Your iPhone sends probes with a different fake MAC each time. You can't reliably track "Alice's phone" across sessions because the identifier changes.

Some devices still leak their real MAC in directed probes (when probing for a specific SSID they've connected to before), but this is increasingly rare. Apple and Google have been tightening randomization with each OS update.

Practical value in 2026

Low. MAC randomization has made probe request tracking unreliable for identifying specific people. You can still detect "some WiFi device is nearby" but not whose it is. Combined with the monitor mode limitations on macOS, this approach is more academic than practical.

4. WiFi fingerprinting (the best option)

This is the most useful laptop-only approach. The idea: every room in your house has a unique pattern of WiFi signal strengths from all visible access points. Your living room might see your router at -45 dBm and the neighbor's at -72 dBm. Your bedroom might see yours at -58 dBm and the neighbor's at -65 dBm. Train a classifier on these patterns and it can tell you which room your laptop is in.

whereami

whereami (~5K GitHub stars) is the easiest implementation. It's a Python CLI that handles scanning, training, and classification.

# Install
pip3 install whereami

# Train: stand in each room and record samples
whereami learn -l "living_room"    # records ~10 scans
whereami learn -l "bedroom"
whereami learn -l "kitchen"
whereami learn -l "office"

# Predict: where am I right now?
whereami predict
# Output: office (0.87 confidence)

# Continuous monitoring
watch -n 5 whereami predict

How it works under the hood

whereami scans all visible access points using the OS WiFi API (CoreWLAN on macOS). Each scan produces a vector: [AP1_rssi, AP2_rssi, ..., APn_rssi]. It trains a Random Forest classifier on labeled vectors. At prediction time, it scans again and asks the classifier which room this vector most closely matches.

The accuracy depends on three things:

  1. Number of visible APs: More APs = more dimensions = better separation. In a typical apartment with 5-10 visible networks, expect 85-90% accuracy. In a dense urban area with 20+ networks, 95%+.
  2. Training quality: Record at least 10 scans per room. Move around the room between scans. Train at different times of day (signal conditions shift slightly).
  3. Room separation: Adjacent rooms with thin walls are hardest to distinguish. Rooms on different floors are easiest.

find3: the multi-device version

find3 (~4.7K stars) extends this concept to track multiple devices. It runs a server that receives fingerprints from multiple clients (phones, laptops, any WiFi device running the scanning agent). The server maintains a location model for each device.

# Server (runs on your local machine or a Raspberry Pi)
docker run -p 8003:8003 schollz/find3

# Client (on the device you want to track)
# find3-cli-scanner is available for macOS, Linux, Windows
find3-cli-scanner -i en0 \
    -server http://localhost:8003 \
    -family "myhouse" \
    -device "macbook" \
    -scantime 3 \
    -learn "office"       # during training
    # omit -learn for prediction mode

Limitations

  • Detects location, not motion. You know which room the laptop is in, not whether someone else just walked in.
  • Requires the laptop to be open and awake.
  • Accuracy degrades when the WiFi environment changes (new router, neighbor moves).
  • Retraining needed periodically (monthly in stable environments, weekly in dynamic ones).

5. Router-based detection

Your router knows exactly which devices are connected, when they connected, when they disconnected, and how much data they're using. If you can query this information programmatically, you have a presence detection system.

The approach

Most routers expose a web interface. Many expose an API. The quality varies enormously by brand:

RouterAPI QualityMethod
OpenWrtExcellentubus JSON-RPC, LUCI REST API
UniFiGoodREST API with auth token
Asus (Merlin)GoodHTTP API, SNMP
TP-LinkPoorScreen-scraping the web UI
ISP-providedTerribleUsually locked down entirely
#!/usr/bin/env python3
"""Query an OpenWrt router for connected devices."""
import requests, json

ROUTER = "http://192.168.1.1"
# Get auth token (replace with your credentials)
auth = requests.post(f"{ROUTER}/cgi-bin/luci/rpc/auth", json={
    "id": 1, "method": "login",
    "params": ["root", "your_password"]
})
token = auth.json()["result"]

# List connected WiFi clients
clients = requests.post(
    f"{ROUTER}/cgi-bin/luci/rpc/sys?auth={token}",
    json={"id": 2, "method": "wifi.getAssocList", "params": []}
)
for client in clients.json().get("result", []):
    print(f"  MAC: {client['mac']}  Signal: {client.get('signal', '?')} dBm")

Practical notes

If you have an OpenWrt or UniFi router, this is the easiest zero-setup option. You don't even need to run code on your Mac continuously. Set up a cron job on the router itself (OpenWrt supports this natively) or poll from your Mac every 30 seconds.

The main limitation is the same as ARP scanning: you're detecting devices, not people. And the update frequency is router-limited. Some routers only refresh their client list every few minutes.

6. BLE scanning

Your MacBook has a Bluetooth radio that can scan for BLE (Bluetooth Low Energy) advertisements. Modern phones, watches, earbuds, and fitness trackers all broadcast BLE advertisements periodically, even when not actively paired to anything.

Working code with bleak

#!/usr/bin/env python3
"""BLE presence scanner. Requires: pip3 install bleak"""
import asyncio
from bleak import BleakScanner
from datetime import datetime

# Known device names or partial names to track
KNOWN_DEVICES = {
    "Alice Watch",     # Apple Watch
    "Galaxy Buds",     # Samsung earbuds
    "Bob AirPods",
}

async def scan_loop():
    print("Scanning for BLE devices...")
    known_nearby = set()

    while True:
        devices = await BleakScanner.discover(timeout=5.0)
        current = set()
        ts = datetime.now().strftime("%H:%M:%S")

        for d in devices:
            name = d.name or "unnamed"
            rssi = d.rssi

            # Check if it's a known device
            for known in KNOWN_DEVICES:
                if known.lower() in name.lower():
                    current.add(known)
                    if known not in known_nearby:
                        print(f"[{ts}] ARRIVED: {known} "
                              f"(RSSI: {rssi} dBm)")

        departed = known_nearby - current
        for d in departed:
            print(f"[{ts}] LEFT: {d}")

        # Summary
        all_count = len(devices)
        tracked = ", ".join(current) if current else "none"
        print(f"[{ts}] {all_count} BLE devices nearby | "
              f"Tracked: {tracked}")

        known_nearby = current
        await asyncio.sleep(10)

if __name__ == "__main__":
    asyncio.run(scan_loop())

RSSI-based distance estimation

BLE RSSI correlates with distance better than WiFi because BLE uses lower transmit power and shorter wavelength. A rough model:

import math

def estimate_distance(rssi, tx_power=-59):
    """Estimate distance in meters from BLE RSSI.
    tx_power: RSSI at 1 meter (typical: -59 for phones)
    Returns distance in meters. Accuracy: +/- 2m indoors."""
    if rssi == 0:
        return -1
    ratio = rssi / tx_power
    if ratio < 1.0:
        return ratio ** 10
    else:
        return 0.89976 * (ratio ** 7.7095) + 0.111

This is a rough estimate. Indoors, walls and reflections make BLE distance estimation accurate to maybe 2-3 meters. Outdoors with line of sight, you can get within 1 meter. Don't rely on this for anything precision-critical.

The MAC randomization problem (again)

Just like WiFi probe requests, BLE advertisements use randomized MAC addresses. You can still identify devices by their advertised name (which many devices include), but unnamed devices can't be tracked across sessions. Apple devices rotate their BLE address every 15 minutes.

Workaround: most personal devices (AirPods, Apple Watch, fitness trackers) include a recognizable device name in their BLE advertisements. Track by name, not by MAC.

7. Why CSI doesn't work on your Mac

This is worth explaining properly because it's the most common question people have when they read about WiFi sensing.

What CSI requires at the hardware level

When your WiFi chip receives a frame, it internally computes the Channel State Information to decode the data. This computation happens in the chip's firmware. The chip uses CSI to figure out how to decode the signal, then throws the CSI away. It never passes it to the operating system because the OS doesn't need it for normal WiFi operation.

To extract CSI, you need to modify the firmware to save and export this data instead of discarding it. This is what the Nexmon project does for Broadcom chips, and what Intel's CSI Tool does for the Intel 5300.

Why macOS blocks this

Three layers of lockout:

  1. Chip support: Modern MacBooks (2020+) use the Broadcom BCM4387. Nexmon has firmware patches for many Broadcom chips (BCM4339, BCM4358, BCM43455c0 in Raspberry Pi), but not the BCM4387. Nobody has reverse-engineered this chip's firmware format.
  2. SIP (System Integrity Protection): Even if you had a BCM4387 firmware patch, macOS prevents loading modified firmware. SIP blocks modifications to system-level components, including WiFi driver firmware. Disabling SIP is possible but Apple increasingly ties system features to SIP being enabled.
  3. Driver integration: Apple's WiFi driver (AirPortBrcmNIC) is tightly integrated with the hardware. It's not the same driver as the open-source brcmfmac used on Linux. A firmware patch that works with the Linux driver won't work with Apple's driver.

What about external USB WiFi adapters?

Theoretically, you could plug in a USB WiFi adapter with a supported Broadcom or Atheros chip and run CSI extraction through it. In practice, macOS driver support for third-party WiFi adapters has been terrible since Apple dropped the kext-based driver model. Most USB WiFi adapters don't work on modern macOS at all, let alone with modified firmware.

If you want CSI on a Mac, the realistic path is: buy an ESP32 ($8), flash ESPectre, and read the data over the network. The ESP32 handles all the RF processing. Your Mac just receives the binary MOTION/IDLE state via WiFi (through Home Assistant, MQTT, or a simple HTTP endpoint).

Security concerns

Every detection method has security and ethical implications. Here's the honest breakdown:

ARP scanning

Risk: Low. You're querying your own network. Anyone on the same LAN could do this. It's equivalent to looking at your router's connected devices page. Don't scan networks you don't own.

RSSI monitoring

Risk: Minimal. You're reading your own WiFi connection's signal strength. No third-party data is captured. This is information your laptop already has and uses internally.

Probe request sniffing

Risk: Moderate. You're passively capturing wireless transmissions from other people's devices. In most jurisdictions, passively receiving radio signals is legal (you're not decrypting anything), but using the data to track individuals may violate privacy laws. GDPR considers MAC addresses personal data. In your own home for your own use, the risk is low. In a public space or for commercial purposes, consult a lawyer.

WiFi fingerprinting

Risk: Low. You're scanning signal strengths of nearby access points. This is the same data your laptop uses to choose which network to connect to. You're not accessing any other network or capturing any traffic.

Router-based

Risk: Low for your own router. You're querying your own equipment's admin API. Accessing someone else's router API without authorization is unauthorized computer access. Don't do that.

BLE scanning

Risk: Moderate. Similar to probe sniffing. You're receiving publicly broadcast BLE advertisements. The data is intentionally transmitted by the devices. But using it to track individuals (especially in public spaces) raises the same privacy concerns as probe sniffing.

General principle

On your own network, in your own home, for your own devices: all of these are fine. The moment you start tracking other people's devices without their knowledge, you're in ethically and potentially legally questionable territory. The technology doesn't care about your intentions. The law does.

What to actually use

If you're reading this and want to set something up today, here's the decision tree:

Goal: know who's home

Use ARP scanning. Map your family members' phone MAC addresses, scan every 30 seconds, done. Ten minutes to set up, 95% reliable.

Goal: know which room you're in

Use whereami. Train it on each room, run it in the background. 30 minutes to train, 85-95% accurate. This is the best no-hardware-required option for room-level awareness.

Goal: detect motion in a room

Use RSSI monitoring as a rough indicator, but honestly, buy an ESP32 for $8 and run ESPectre. The laptop-only RSSI approach is a proof of concept, not a reliable system. CSI-based detection on an ESP32 is 99%+ accurate. RSSI monitoring on a laptop is 60-75%.

Goal: detect nearby people (not on your network)

Use BLE scanning with bleak. It'll pick up phones, watches, earbuds, and fitness trackers within 10-15 meters. You won't know whose they are (MAC randomization), but you'll know someone's nearby.

Goal: build a real presence detection system

Combine ARP scanning (who's home) + whereami (which room) + an ESP32 with ESPectre (is someone actually in this room right now). Layer the methods. Each covers a gap the others miss. Total hardware cost: $8 for the ESP32. Total software cost: $0.
