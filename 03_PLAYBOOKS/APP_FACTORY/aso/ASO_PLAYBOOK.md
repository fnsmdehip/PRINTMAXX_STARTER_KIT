## Title optimization (30 characters max)

### Rules

1. **Primary keyword first** - Most important keyword in first 10 chars
2. **Brand name** - Include app name for recognition
3. **No keyword stuffing** - Apple rejects obvious stuffing
4. **Character count** - Exactly 30 chars max for iOS, 30 for Google Play title

### Formulas

**Formula 1: Brand + Primary keyword**
```
[Brand Name] - [Primary Keyword]
```

**Formula 2: Keyword with brand**
```
[Keyword]: [Brand Name]
```

**Formula 3: Action-oriented**
```
[Brand] - [Action] [Object]
```

## Subtitle optimization (30 characters max)

### Rules

1. **Secondary keywords** - Don't repeat title keywords
2. **Benefit-focused** - What user gets
3. **Complement title** - Fill keyword gaps
4. **No punctuation spam** - One comma max

### Subtitle formulas

**Formula 1: Benefit statement**
```
[Benefit] for [Audience]
```

**Formula 2: Feature pair**
```
[Feature 1] & [Feature 2]
```

**Formula 3: Action result**

```
[Action] to [Result]
```

## Keyword field strategy (100 characters - iOS only)

### Rules

1. **No spaces after commas** - Wastes characters
2. **No duplicates** - Don't repeat title/subtitle words
3. **Singular forms** - Apple indexes plurals automatically
4. **No competitor names** - Rejected
5. **No "app" or "free"** - Wasted, already indexed

### Keyword prioritization

**Tier 1: Include always**
- Branded misspellings
- Core function keywords
- Niche-specific terms

**Tier 2: Include if space**
- Benefit keywords
- Related actions
- Audience terms

**Tier 3: Rotate monthly**

- Trending terms

## Description optimization

### Structure (4000 chars max iOS, Google Play varies)

```

[What it does - 2-3 sentences]

KEY FEATURES:
- [Feature 1 with keyword]
- [Feature 2 with keyword]
- [Feature 3 with keyword]
- [Feature 4 with keyword]
- [Feature 5 with keyword]

[Social proof - download count, ratings, press mentions]

[How it works - 2-3 sentences]

[Who it's for - target audience]

[Call to action]

[Subscription disclosure - required for Apple]
```

### Description best practices

1. **Keywords in first 250 chars** - Google indexes this heavily
2. **Feature bullets** - Easy to scan
3. **Social proof** - Numbers, quotes, press
4. **Subscription clarity** - Required, put at end
5. **No competitor mentions** - Rejected
6. **Update regularly** - Fresh content helps ranking

## Screenshot A/B testing

### Screenshot requirements

**iOS:**
- 6.7" (iPhone 15 Pro Max): 1290 x 2796
- 6.5" (iPhone 11 Pro Max): 1242 x 2688
- 5.5" (iPhone 8 Plus): 1242 x 2208
- iPad Pro 12.9": 2048 x 2732

**Google Play:**
- Phone: 1080 x 1920 minimum
- 7" tablet: 1200 x 1920
- 10" tablet: 1800 x 2560

### Screenshot slots (optimize in order)

1. **Screenshot 1** - Primary benefit/hook (80% of views)
2. **Screenshot 2** - Core feature demo
3. **Screenshot 3** - Secondary feature
4. **Screenshot 4** - Social proof or results
5. **Screenshot 5** - Additional feature
6. **Screenshot 6+** - Nice to have

### A/B test framework

**Test 1: Hook copy**
- Variant A: Feature-focused ("Block phone until you pray")
- Variant B: Benefit-focused ("Start mornings with faith")
- Variant C: Problem-focused ("Stop scrolling, start praying")

**Test 2: Visual style**
- Variant A: Device mockup with UI
- Variant B: Lifestyle imagery with text overlay
- Variant C: Icon/illustration focused

**Test 3: Social proof**
- Variant A: Star rating prominently displayed
- Variant B: User count ("50,000+ users")
- Variant C: Testimonial quote

### Test duration and sample size

- Minimum: 7 days
- Minimum impressions: 5,000 per variant
- Statistical significance: 95% confidence
- Tools: App Store Connect (iOS), Google Play Console

### Screenshot copy formulas

**Formula 1: Problem-solution**
```
[Problem statement]
[Arrow or visual separator]
[Solution with app]
```

**Formula 2: Before-after**
```
Before: [Bad state]
After: [Good state with app]
```

**Formula 3: Feature-benefit**
```
[Feature name]
[What it does for user]
```

## Icon testing

### Icon requirements

**iOS:**
- 1024 x 1024 (App Store)
- No transparency
- No rounded corners (system applies)

**Google Play:**
- 512 x 512
- 32-bit PNG with alpha
- Can have transparency

### Icon best practices

1. **Simple** - Recognizable at 60x60 pixels
2. **Unique** - Different from competitors
3. **Brand-aligned** - Matches app personality
4. **No text** - Text unreadable at small sizes
5. **One focal point** - Single element dominates

### Icon A/B test framework

**Test 1: Color scheme**
- Variant A: Brand primary color
- Variant B: Contrasting color
- Variant C: Gradient

**Test 2: Imagery**
- Variant A: Abstract symbol
- Variant B: Literal representation
- Variant C: Mascot/character

**Test 3: Style**
- Variant A: Flat design
- Variant B: 3D/dimensional
- Variant C: Outlined/minimalist

## Update frequency strategy

### Why updates matter for ASO

1. **Freshness signal** - Stores favor active apps
2. **Review reset opportunity** - Prompt new reviews
3. **Keyword refresh** - Update metadata
4. **Feature iteration** - Improve conversion

### Update cadence

| Phase | Frequency | Focus |
|-------|-----------|-------|
| Launch (0-30 days) | Weekly | Bug fixes, quick iterations |
| Growth (30-90 days) | Bi-weekly | Features, ASO experiments |
| Mature (90+ days) | Monthly | Maintenance, seasonal updates |

### What to update

**Every update:**
- Bug fixes (even minor)
- Performance improvements
- What's New section refreshed

**Monthly:**
- Screenshot A/B tests
- Keyword field optimization
- Description refinements

**Quarterly:**
- Major feature additions
- Icon refresh tests
- Pricing experiments

### Version numbering strategy

```
Major.Minor.Patch
1.0.0 - Initial launch
1.0.1 - Bug fixes
1.1.0 - Minor feature
1.2.0 - Minor feature
2.0.0 - Major redesign
```

### What's New best practices

```
Version 1.2.0 - Better mornings, fewer bugs

- Morning reminder improvements
- Faster app startup
- Fixed notification timing issue

Love PrayerLock? Leave a review!
```

Keep it:
- Short (3-5 bullets)
- User-focused (benefits, not technical)
- Ending with review prompt

## Review generation strategy

### When to prompt for reviews

**Good times:**
- After completing a goal (prayer, steps, etc.)
- After 3+ successful sessions
- After positive interaction (achievement unlocked)
- 7 days after download

**Bad times:**
- On first open
- After error or crash
- During core task
- Before user gets value

### StoreKit 2 implementation (iOS)

```swift
// Request review after positive moment
if sessionCount >= 3 && hasCompletedGoal {
    SKStoreReviewController.requestReview()
}
```

**Apple limits:** 3 requests per 365 days per device

### Google Play In-App Review

```kotlin
// Similar timing logic
val manager = ReviewManagerFactory.create(context)
val request = manager.requestReviewFlow()
request.addOnCompleteListener { task ->
    if (task.isSuccessful) {
        val reviewInfo = task.result
        manager.launchReviewFlow(activity, reviewInfo)
    }
}
```

### Responding to reviews

**Negative reviews (1-2 stars):**
- Respond within 24 hours
- Apologize for experience
- Ask for specifics
- Offer to help via email
- Update response when fixed

**Positive reviews (4-5 stars):**
- Thank them
- Highlight what they mentioned
- Keep it short

**Example negative response:**
```
Hi [Name], sorry to hear about this issue. We'd love to help fix this for you. Can you email us at support@prayerlock.app with your device model? We'll get this sorted quickly.
```

**Example positive response:**
```
Thank you! So glad PrayerLock is helping you start mornings with faith. God bless!
```
