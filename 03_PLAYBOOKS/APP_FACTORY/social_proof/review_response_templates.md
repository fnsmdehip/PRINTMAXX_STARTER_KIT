# Review response templates

Templates for responding to app store reviews and feedback.

---

## Why respond to reviews

- Shows you're actively maintaining the app
- Can convert negative reviewers to positive ones
- Provides context for potential users reading reviews
- Boosts app store ranking signals
- Gives you product feedback

---

## Positive review responses (5 variations)

### 1. Simple thank you

Use for: Short positive reviews without specific details.

```
Thanks for the kind words, [Name]! Glad [App Name] is working well for you.
```

### 2. Feature acknowledgment

Use for: Reviews that mention a specific feature.

```
Thanks [Name]! Happy to hear [specific feature] is saving you time. That was one of the features we spent the most time on.
```

### 3. Usage acknowledgment

Use for: Reviews that describe how they use the app.

```
Thanks for sharing how you use [App Name], [Name]. Love hearing about [their specific use case]. Let us know if there's anything else we can add to help with that.
```

### 4. Power user recognition

Use for: Detailed positive reviews from clearly engaged users.

```
Really appreciate the detailed review, [Name]. You've clearly put the app through its paces. If you ever have feature ideas, we're always listening - [contact method].
```

### 5. Update mention

Use for: Positive reviews after a recent update.

```
Thanks [Name]! We just released [recent update/feature] last week. Glad you're finding it useful. More improvements coming soon.
```

## Negative review responses (5 variations)

### 1. Bug acknowledgment

Use for: Reviews describing a legitimate bug.

```
Sorry about this, [Name]. This sounds like a bug we need to fix. Can you email [support email] with details? We'll get it sorted and follow up with you directly.
```

### 2. Feature request acknowledgment

Use for: Negative reviews that are really feature requests.

```
Thanks for the feedback, [Name]. We don't have [requested feature] yet, but it's on our list. I'll bump the priority based on your feedback. We'll update when it's ready.
```

### 3. Misunderstanding clarification

Use for: Negative reviews based on misunderstanding how the app works.

```
Thanks for the feedback, [Name]. It sounds like [feature] might not be obvious. Try [brief instruction]. If that doesn't help, email [support email] and we'll walk you through it.
```

### 4. Update notification

Use for: Reviews about issues fixed in a recent update.

```
Hi [Name], we fixed this in our latest update (version [X]). Can you update and try again? If the issue persists, email [support email]. Happy to help directly.
```

### 5. Service recovery

Use for: Frustrated reviews from users who had a bad experience.

```
Sorry [App Name] didn't meet your expectations, [Name]. I'd like to make this right. Can you email me directly at [email]? I'll personally look into what went wrong.
```

## Feature request responses

### Planned feature

```
Good news - [requested feature] is on our roadmap. No exact date yet, but it's coming. Thanks for the nudge, [Name].
```

### Under consideration

```
Thanks for the suggestion, [Name]. We've heard this from a few users now. Adding it to our list for consideration. No promises on timeline, but we're listening.
```

### Not planned (polite decline)

```
Thanks for the feedback, [Name]. We considered [requested feature] but decided to focus on [current direction] instead. Appreciate you taking the time to share your thoughts.
```

### Already exists

```
Good news, [Name] - you can already do this! Go to [location] and [brief instruction]. Let us know if you need help finding it.
```

## Bug report responses

### Investigating

```
Thanks for reporting this, [Name]. We're looking into it now. Can you email [support email] with your device model and OS version? That'll help us track it down faster.
```

### Fixed in next update

```
Found the issue, [Name]. Fix is ready and will be in our next update (submitting this week). Thanks for helping us catch this.
```

### Cannot reproduce

```
Thanks for reporting, [Name]. We're having trouble reproducing this on our end. Can you email [support email] with more details? Screenshots or screen recordings help a lot.
```

### User-side issue

```
Hi [Name], this sounds like it might be related to [device/OS/settings]. Try [troubleshooting step]. If that doesn't work, email [support email] and we'll dig deeper.
```

### Known issue with workaround

```
Thanks for reporting, [Name]. This is a known issue we're working on. In the meantime, you can [workaround]. Fix coming in a future update.
```

## Response guidelines

### Do

- Respond within 24-48 hours
- Use reviewer's name (if available)
- Be specific and helpful
- Provide contact method for follow-up
- Thank them for feedback (even negative)
- Keep responses under 100 words
- Update responses if you fix the issue

### Don't

- Get defensive or argumentative
- Make promises you can't keep
- Copy/paste identical responses
- Ignore legitimate complaints
- Ask them to change their rating
- Offer incentives for review changes
- Disclose private user information

---

## Response timing

**Negative reviews:** Respond within 24 hours. Speed matters.

**Positive reviews:** Respond within 48-72 hours. Less urgent but still good practice.

**Review with questions:** Respond ASAP. They're asking for help.

**Old reviews:** Still worth responding if the issue is now fixed. Say "Hi [Name], checking back in. We fixed this in version [X]. Hope it's working better for you now."

---

## Tracking responses

Keep a log of:
- Review date
- Rating (1-5 stars)
- Main complaint/praise
- Your response
- Date responded
- Any follow-up outcome (did they update review?)

Use this to:
- Identify recurring issues
- Measure response effectiveness
- Track review sentiment over time
- Prioritize bug fixes and features
