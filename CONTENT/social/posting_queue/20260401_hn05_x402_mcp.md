## POST 5 — Show HN

**Title:** Show HN: x402 micropayment wrappers for MCP servers — AI agents pay per API call in USDC

**Body:**
x402 is a new protocol (Zuplo/Coinbase, Q1 2026) that implements HTTP 402 Payment Required with USDC micropayments. The use case: AI agents autonomously pay for API calls without human intervention.

The problem: 11,000+ MCP servers exist. Almost none are monetized. Builders maintain infrastructure for free. Users get unreliable service because free tools have no sustainability model.

The approach: wrap existing MCP server endpoints with x402 payment headers. Agent makes request, server returns 402 + price, agent wallet auto-pays $0.001-0.01 USDC, server returns data.

What I'm building:
- Middleware that adds x402 to any existing MCP server with one config change
- Pricing dashboard for monitoring per-endpoint revenue
- USDC settlement to any EVM-compatible wallet

The math on my own system: 529 automation scripts exist (scraping, analysis, scoring, content generation). Wrapping even 10% as x402 endpoints at 100 calls/day x $0.005 = $26.50/day from scripts that already run.

The broader implication: autonomous B2B commerce at the agent level. Agent A (company 1) needs data from Agent B (company 2). x402 handles payment. No contracts, no invoicing, no accounts payable department.

Protocol is open. Less than 5% adoption across the MCP ecosystem. Tooling is sparse.

Looking for feedback on:
1. Is $0.001-0.01 per call the right price range for MCP tools?
2. What endpoints would you pay for if they had x402?
3. Any concerns about USDC settlement speed for real-time agent interactions?

## POST 6 — Show HN

**Title:** Show HN: I score my app portfolio like a quant fund — 7-dimension composite scoring with automated kill/scale rules

**Body:**
I have 8 iOS apps in various stages. Instead of deciding what to work on by gut feel, I built a scoring system borrowed from quantitative portfolio management.

Every app gets a weekly composite score across 7 dimensions:
1. Revenue potential (TAM x conversion rate x price point)
2. Speed to first dollar (how quickly can this generate revenue)
3. Automation potential (can AI agents maintain it with minimal human input)
4. Downside risk (regulatory, App Store rejection, competitor moat)
5. Competition density (how many apps target the same niche)
6. Synergy (does this app feed leads/content/data to other apps in portfolio)
7. Capital efficiency (revenue per hour of dev time)

Kill rules are non-negotiable:
- Below $100 MRR after 60 days = kill
- Below 500 users after 90 days = kill
- 20%+ sustained growth at $500+ MRR = double down, shift resources from bottom performers

The script runs weekly via cron, pulls actual revenue data, scores everything, and outputs a recommended action for each app: kill, maintain, scale, or boost.

This is the same math hedge funds use for position sizing: the Kelly Criterion applied to software development. Single app is a concentrated bet. Portfolio with kill discipline is a diversified strategy.

Written in Python. About 400 lines. Happy to discuss the scoring methodology or the kill rule calibration.
