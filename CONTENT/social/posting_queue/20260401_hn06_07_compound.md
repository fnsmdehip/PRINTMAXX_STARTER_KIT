## HN 06 — Show HN: Agent Swarm Performance Review

**Title: Show HN: I built a self-evaluating AI agent swarm. After 48 cycles it fired half itself**

**URL:** (self post)

**Text:**

I run 33 AI agents on cron jobs. Every 2 hours a "swarm brain" agent reviews all others: reads their latest reports, scores utility, decides promote/maintain/kill.

After 48 cycles, the brain reduced the productive fleet from 25 to 12. The rest were generating reports nobody consumed.

Interesting findings from the evaluation data:

1. Infrastructure agents (health monitoring, deployment tracking) consistently scored highest. They prevent failures, which is measurable.

2. Intelligence agents (competitor scanning, lead qualification) scored well ONLY when their output fed directly into an action agent. Standalone intel with no consumer scored near zero.

3. Content generation agents scored poorly because the distribution channel wasn't connected (no social accounts set up). Perfectly good output, no way to ship it. The brain correctly identified this as a system constraint, not an agent failure.

4. The data janitor agent was the most surprising performer. It found that a scraper had a merge() bug producing 99.9% duplicate rows. 2,807 of 2,810 rows identical. Every agent downstream was reasoning over garbage data.

Architecture: Claude Code on cron via launchd. Filesystem is the memory layer (not vector DB). Each agent reads state files, writes reports, updates shared JSONs. The brain reads all outputs and writes decisions to a JSONL log.

The counterintuitive lesson: the most valuable meta-agent isn't the one that orchestrates work. It's the one that kills unnecessary work.

Curious if others have built self-evaluating multi-agent systems and what patterns emerged.

## HN 07 — surge.sh robots.txt discovery

**Title: Surge.sh free tier silently blocks all search engine crawling via robots.txt**

**URL:** (self post)

**Text:**

After deploying 388 sites to surge.sh's free student tier over 2 months, I discovered that surge injects `Disallow: /` into robots.txt at the CDN level.

This cannot be overridden by uploading your own robots.txt. The CDN serves their version regardless of what's in your deployment.

Impact: all SEO work is nullified. Sitemaps, meta tags, structured data, canonical URLs -- none of it matters if crawlers can't index the pages.

Verified with: `curl -s [site].surge.sh/robots.txt`

Surge Plus ($13/mo) removes the restriction. Netlify and Cloudflare Pages free tiers don't have this limitation.

Documenting here because I couldn't find this clearly stated in surge.sh's docs or pricing page. It's not in the FAQ. The only way to discover it is to check robots.txt on a deployed site.

If you're using surge for anything that depends on organic search traffic, check your robots.txt now.
