152 websites. $0 hosting costs. $0 revenue.

Here's the real story.

Over the past several months I built and deployed 152 websites across surge.sh. Cost to host all of them: $0. Some of them look genuinely good. A few of the tools actually solve real problems.

None of them make money yet. Here's why.

**What went right:**

Deployment speed is real. surge.sh deploys a site in under 10 seconds. No config, no server provisioning, no DNS headaches. You drop a folder and it's live. I went from zero to 152 live URLs faster than most people finish setting up a single VPS.

The PWA architecture held up. Service workers, web app manifests, offline-first storage with localStorage and IndexedDB. No backend means no server bill, no database costs, no auth to maintain. 13 streak-tracking apps running right now at $0/mo infrastructure cost total.

The stack is right. Vanilla JS, static HTML, surge.sh, no dependencies that can break or go paid.

**What went wrong:**

I built the wrong thing in the wrong order.

I have 7 apps fully built. I have 23 free SaaS tools live. I have 152 deployed pages. What I don't have is a Gumroad account that's verified, a Stripe account set up, or a monetization layer attached to any of it.

That's a systems problem, not a building problem. I optimized for shipping when I should have been optimizing for the full pipeline: build, then wire up payments, then distribute, then build the next thing.

The other problem: I was building when I should have been distributing. The ratio was probably 90% building, 10% distributing. It should be closer to 50/50.

**The actual lesson:**

Building is the easy part. Any decent developer with Claude can deploy 152 sites. The hard part is the 15 steps after deployment: adding a payment link, writing the cold email, posting the Show HN, DMing the niche community, collecting the first email address.

I have 152 proof points that the infrastructure works. Now the job is turning them into 152 revenue experiments.

If you're further along in the monetization side of this, I'd genuinely like to know what the first $100 looked like for you.

sitescore-app.surge.sh

BODY:
I kept paying for Lighthouse-adjacent tools that were wrapping free public APIs. Decided to build the same thing myself in a weekend.

SiteScore calls 4 APIs directly from the browser: Google PageSpeed Insights (performance score, Core Web Vitals), SSL Labs (certificate grade), SecurityHeaders.io (response header audit), and Mozilla Observatory (security score). Weights them into a single 0-100 score with a per-category breakdown. No backend. No server. Static HTML and vanilla JS hosted on surge.sh. Your URL never touches my infrastructure because there is no infrastructure.

The whole audit takes 8-12 seconds on a cached domain. SSL Labs adds 30-60 seconds on an uncached one because they run a full handshake analysis -- that wait is on their end, not mine.

Known limitations:
- SecurityHeaders rate-limits at 10 requests/hour per IP. Fine for personal use, breaks for any automated pipeline.
- SSL Labs queues uncached domains. If you're the first person to check a domain today, expect to wait.
- PageSpeed Insights results vary by time of day depending on Google's sampling. Not a bug in my code.
- No historical tracking. Every audit is stateless. I store nothing.
- No PDF export or sharing links yet.

The broader observation: most "premium" site audit tools charge $50-200/mo for a UI layer on top of the same free APIs. The APIs themselves are well-documented and freely available. The value these tools add is mostly UX, historical data, and team sharing -- none of which you need for a one-off audit.

Stack: HTML, CSS, vanilla JS, Fetch API for the calls, surge.sh for hosting. No npm, no build step, no framework. The whole thing is one HTML file and one JS file.

Source is not yet public but I'm considering it. Happy to answer questions about the API integration approach or the weighting logic for the score.

TITLE:
Show HN: 13 offline-first streak PWAs built from one template -- open-sourcing the base

URL:
quranstreak-app.surge.sh

BODY:
I wanted to see how far you could push the no-backend PWA approach before it breaks. The answer is: further than I expected.

13 apps. All streak trackers. Different micro-communities: Quran readers, Torah learners, Sikh practitioners, Catholics, Lutherans, Anglicans, Methodists, meditators, coders, runners, artists, language learners, journal keepers. All live on surge.sh. Total infrastructure cost: $0/mo.

The architecture is the same across all 13:

Service worker with a cache-first strategy. On first load, the SW caches all assets (HTML, CSS, JS, icons). From day 2 on, the app loads from cache and never touches the network unless you force a refresh. This means it works on a plane, in a rural area with no signal, in the subway.

State lives in localStorage. Streak history (date, completion boolean, optional note) lives in IndexedDB. Nothing leaves the device. No account creation prompt. No "sign in to sync" friction. You open it, you mark your streak, you close it.

Web App Manifest gives you the installable PWA experience on Android (install to home screen, fullscreen mode, custom splash). iOS is limited -- no service worker push notifications, no "add to home screen" prompt, you have to tell users to do it manually. That's a real limitation.

The deploy loop:

```bash
for app in quran torah sikh catholic lutheran anglican methodist meditation coding running art language journal; do
  surge ./apps/$app ${app}streak-app.surge.sh
done
```

13 deploys in about 4 minutes total.

The template I'm open-sourcing has: the service worker boilerplate, the Web App Manifest template, the localStorage/IndexedDB abstraction layer, the streak calculation logic (handles timezone edge cases), and a config file where you swap out the branding. To spin up a new streak app, you edit the config file and run the deploy command. About 15 minutes from config to live URL.

The actual question I'm sitting with: what's the adoption ceiling for offline-only apps with no cross-device sync? The apps are live. I don't have good retention data yet beyond 30-day installs.
