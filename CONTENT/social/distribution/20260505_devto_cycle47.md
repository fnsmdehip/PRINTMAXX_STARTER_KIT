**Title:** AES-256-GCM Client-Side Encryption with Web Crypto API — No Server Required

**Body:**

I built cnsnt — an offline consent form manager. The core requirement was: all data must be encrypted on the client before any storage or export, with no server-side component.

This meant implementing AES-256-GCM entirely in the browser using the Web Crypto API. Here's how I approached the three hard parts.

---

### Why Web Crypto API Over node-crypto

The app runs as a PWA with no backend. node-crypto doesn't exist in the browser. Web Crypto API is the native browser cryptography library — available in all modern browsers, runs in service workers, and uses hardware acceleration where available.

Tradeoff: Web Crypto API is callback/Promise-based with a different interface than node-crypto. Takes adjustment if you're used to the synchronous node API.

---

### Part 1: Key Derivation from a User PIN

The core problem: derive an encryption key from a user-supplied PIN without storing the PIN or any server-side secret.

I used PBKDF2 with the following parameters:

```javascript
async function deriveKey(pin, salt) {
  const encoder = new TextEncoder();
  
  // Import the PIN as a raw key
  const baseKey = await crypto.subtle.importKey(
    'raw',
    encoder.encode(pin),
    { name: 'PBKDF2' },
    false,
    ['deriveKey']
  );
  
  // Derive AES-GCM key with 100,000 iterations
  return crypto.subtle.deriveKey(
    {
      name: 'PBKDF2',
      salt: encoder.encode(salt),
      iterations: 100000,
      hash: 'SHA-256',
    },
    baseKey,
    { name: 'AES-GCM', length: 256 },
    false,
    ['encrypt', 'decrypt']
  );
}
```

100,000 iterations is OWASP's current minimum recommendation for PBKDF2-SHA256. The salt is stored alongside the encrypted data (salts aren't secret — they prevent rainbow table attacks by making each encryption unique).

---

### Part 2: AES-256-GCM Encryption

GCM mode was chosen over CBC because GCM provides authenticated encryption — it produces an authentication tag that verifies both the data integrity and that the correct key was used. No separate HMAC needed for the encryption layer.

```javascript
async function encryptData(data, key) {
  const encoder = new TextEncoder();
  const iv = crypto.getRandomValues(new Uint8Array(12)); // 96-bit IV for GCM
  
  const encrypted = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    encoder.encode(JSON.stringify(data))
  );
  
  // Bundle IV + ciphertext for storage
  return {
    iv: Array.from(iv),
    ciphertext: Array.from(new Uint8Array(encrypted)),
  };
}

async function decryptData(encryptedBundle, key) {
  const iv = new Uint8Array(encryptedBundle.iv);
  const ciphertext = new Uint8Array(encryptedBundle.ciphertext);
  
  const decrypted = await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv },
    key,
    ciphertext
  );
  
  return JSON.parse(new TextDecoder().decode(decrypted));
}
```

The GCM authentication tag is appended to the ciphertext automatically by the Web Crypto API — you don't handle it separately. If decryption fails due to a wrong key or tampered data, `decrypt()` throws a `DOMException`.

---

### Part 3: HMAC-SHA-256 File Integrity

The GCM tag covers the encryption layer. But I also wanted integrity checks at the file level — to detect corruption or tampering of the stored bundle before attempting decryption.

```javascript
async function generateHMAC(data, key) {
  const encoder = new TextEncoder();
  
  const hmacKey = await crypto.subtle.importKey(
    'raw',
    await crypto.subtle.exportKey('raw', key),
    { name: 'HMAC', hash: 'SHA-256' },
    false,
    ['sign', 'verify']
  );
  
  const signature = await crypto.subtle.sign(
    'HMAC',
    hmacKey,
    encoder.encode(JSON.stringify(data))
  );
  
  return Array.from(new Uint8Array(signature));
}
```

The HMAC is computed over the encrypted bundle and stored alongside it. On load, verify the HMAC before attempting decryption — if it fails, the file has been tampered with or corrupted.

---

### Storage Pattern

Final stored structure in localStorage:

```json
{
  "version": 1,
  "salt": [array of bytes],
  "bundle": {
    "iv": [array of bytes],
    "ciphertext": [array of bytes]
  },
  "hmac": [array of bytes]
}
```

Load sequence:
1. Read stored object
2. Verify HMAC (fail fast if tampered)
3. Derive key from user PIN + stored salt
4. Decrypt bundle (GCM tag verification is implicit in decrypt())
5. Parse decrypted JSON

---

### What I'd Do Differently

The current implementation re-derives the key on every access. For a desktop app, caching the derived key in memory during the session would improve UX without meaningfully reducing security. I haven't implemented this yet.

Also: the HMAC key is derived from the encryption key. Ideally these would be separate keys derived from separate PBKDF2 operations. The current implementation is sufficient for the threat model (no server, local tamper detection) but could be tightened.
